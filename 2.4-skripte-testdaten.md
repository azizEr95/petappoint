### 1. seed-static.ts

**Typ:** Seed (Persistente Basisdaten)
**Ausführung:** `npm run seed:static` (backend root) oder `npm --prefix ./backend run seed:static` (root)
**Zeitpunkt:** Einmalig nach fresh DB setup oder manueller reset

**Was wird erstellt:**
- **15 Veterinary Practices** (Berlin, Hamburg, München, Köln, Frankfurt, Stuttgart, Düsseldorf, Leipzig, Dresden, Hannover, Nürnberg, Bremen, Essen, Dortmund, Bonn)

1. **Foundation Phase:** Services, Animal Types, Races, Groups einrichten (idempotent)

**Idempotenz:**
- Alle Entities werden via email/eindeutiger ID gecheckt
- Existierende Daten werden übersprungen
- Confirmation codes + verification aktiviert
- **Passwort für alle Accounts:** `Vetilib123!` (auto-hashed via Prisma extension)

**Besonderheiten:**
- Veterinarians als Person mit zusätzlicher Veterinarian-Relation

### 2. seed-dynamic.ts

**Typ:** Seed (Datumsrelative Termine)
**Ausführung:** `npm run seed:dynamic` (backend root) oder `npm --prefix ./backend run seed:dynamic` (root)
**Zeitpunkt:** Periodisch zum Aktualisieren der Termine, oder nach static seed

**Was wird erstellt:**
- **~4500 Appointments** (destruktiv: löscht alte appointments/links)
- **Past appointments (-14 bis -1 Tage):** ~1000 total, 30-40% mit animal linkage
- **Future appointments (+1 bis +28 Tage):** ~3500 total, 25% booked (mit animal), 75% available (frei)
- **Appointment-Service Links:** Services basierend auf vet specializations

**Logik:**
1. **Fetch:** Alle practices, vets, services, animals aus DB
2. **Delete:** Alte appointments + appointment-service links (transaction)
3. **Past Appointments:**
   - Loop -14 bis -1 Tage
   - Weekday: 5-8 appointments/tag
   - Weekend: 2-4 appointments/tag
   - Zufällige Uhrzeiten 8:00-17:59 (0 oder 30 min)
   - Duration: 30-75 Minuten (randomisiert)
   - 35% haben animal linkage
4. **Future Appointments:**
   - Loop +1 bis +28 Tage (Praxen 11-15: start at +15)
   - Weekday: 8-12 appointments/tag
   - Weekend: 4-6 appointments/tag
   - Zufällige Uhrzeiten 8:00-17:59
   - 25% haben animal linkage (booked), 75% frei
5. **Batch Create:** Appointments in 1000-batches
6. **Service Links:**
   - Available appointments: ALL vet services verlinkt
   - Booked appointments: ONE random vet service verlinkt

**Destruktiv:** ⚠️ **WICHTIG:** Löscht alle existing appointments + service links! Nicht für production data!

**Summary Output:**
```
🌱 Seeding dynamic appointments...
📅 Generating past appointments (-14 to -1 days)...
📅 Generating future appointments (+1 to +28 days)...
📝 Creating ~4500 appointments in batches...
🔗 Linking appointments to services...
✅ Dynamic seeding complete!
   📅 Total appointments: ~4500
   📆 Past appointments: ~1000 (booked: ~350)
   📆 Future appointments: ~3500 (available: ~2625, booked: ~875)
```

   - appointments, services
   - addresses, countries
   - confirmationcodes, invoices, reviews
   - junction tables (person_has_animal, veterinary_has_service, veterinary_has_specialization, etc.)

### Test & Integration Scripts

**run-integration.ts** (`backend/scripts/run-integration.ts`)
- Startet Docker DB wenn nicht running
- Wartet auf DB port 5432 (via wait-for-it.sh)
- Führt Prisma migration aus
- Startet Jest tests

**run-integration.sh** (`backend/scripts/run-integration.sh`)
- Shell wrapper um run-integration.ts
- Optional: `--ui` flag für Vitest UI

**setenv.sh** (`backend/scripts/setenv.sh`)
- Exportiert `.env` variables
- Sourced by run-integration.sh für environment setup

**wait-for-it.sh** (`backend/scripts/wait-for-it.sh`)
- TCP port availability checker (BSD nc implementation)
- Wartet bis PostgreSQL auf port 5432 antwortet
- Timeout: default 15s

```bash
# 1. Install alle dependencies
npm install

# 2. Build shared package
npm run build
```

**Result:** Alle workspaces installiert, shared gebaut

### Flow 2: Development Startup (Complete Stack)

```bash
# OPTION A: One-command startup
npm run vetilib

# Oder OPTION B: Manual steps
npm run vetilib:detached        # Start Docker DB
npm run migrateDB               # Prisma migration
npm run seed                    # seed:all (static + dynamic)
npm run startBackendFrontend    # Backend + Frontend parallel

# OPTION C: Start einzelne Services
npm run startBackend            # Port 3000
npm run startFrontend           # Port 3001
```

**Result:**
- PostgreSQL DB on localhost:5432
- Backend on localhost:3000
- Frontend on localhost:3001
- ~5 seconds startup time für Seed (~4500 appointments)


**Result:** Frische DB mit allen static + dynamic testdata

### Flow 5: Backend Testing

```bash
cd backend

# OPTION B: Mit UI
npm run test:ui                 # run-integration.sh --ui
```

---

### Flow 6: E2E Testing (Recommended)

```bash
# OPTION A: Full E2E (only in dev, not CI-optimized)
npm run e2e-tests               # npm-run-all mit startDB (zu langsam)

# OPTION B: CI Mode (Recommended)
npm run e2e-tests:ci            # cd e2e-tests && playwright test
# Startet DB, Backend, Frontend automatisch
# Single worker, 1 retry, erzeugt HTML + JUnit XML reports
```

**Before E2E Tests:**
- DB + Backend müssen laufen ODER Playwright startet sie via playwright.config.ts
- seed-e2e-tests.ts wird automatisch via globalSetup executed
- Tests nutzen fixed test data (nicht ändern!)

**Result:** E2E tests pass mit isolated, reproducible test data

## Architektur-Referenz

### Seed System: 2-Tier Architecture

**Motivation:** Früher single seed script → Probleme:
- Double-hashing passwords
- Expired dates (relativ zu seed-time)
- Keine idempotenz
- Ungleiche appointment distribution

**Lösung: Separate static & dynamic seeds**

```
seedDyn├─► Destruktiv (delete-recreate pattern)
  ├─► Date-relative logic (appointments shift daily)
  ├─► Nur für appointments + service links
  └─► Anwendungsfall: Refresh appointments für neue "heute"

seedStat├─► Idempotent (email checks, skip if exists)
  ├─► Fixed data: practices, persons, animals, vets
  ├─► Basis für dynamic seed (vets, services, animals)
  └─► Anwendungsfall: Einmalig nach fresh DB
```

**Orchestration:** seed-all.ts → static then dynamic

### Prisma Extensions: Auto-Hashing

**singletonPC.ts:**
```typescript
// Client extension für passwords:
$allOperations → if (password field) → bcrypt.hash(password, 10)
// Automatisch bei create/update operations
```

**Benefit:** Keine manuellen hashing scripts nötig

### E2E Test Strategy

**Isolation:** Separate database oder clean seeds vor tests

**seed-e2e-tests.ts → globalSetup (playwright.config.ts)**
```
playwright test
  ├─► globalSetup executed → seed-e2e-tests.ts runs
  ├─► Wipe DB
  ├─► Create minimal test data (2 practices, 3 vets, 4 persons, appointments)
  └─► Tests run against fresh, known state
```

**WICHTIG:** E2E seed data **nicht ändern** (Tests hängen davon ab!)

## Troubleshooting

### "npm run seed:static" fails with "Germany not found"

**Ursache:** `init.sql` konnte nicht ausführt werden (DB hat keine countries)

**Lösung:**
```bash
npm run stop                    # Stop Docker
npm run vetilib:detached        # Restart (fresh init)
npm run migrateDB               # Prisma migration
npm run seed                    # Try again
```

### Docker port 5432 already in use

**Ursache:** Alte DB container läuft noch

**Lösung:**
```bash
docker ps -a                    # List all containers
docker stop <container-id>      # Stop container
docker rm <container-id>        # Remove container
npm run vetilib:detached        # Start fresh
```

### Appointments nie älter als -14 Tage

**Design:** seed-dynamic.ts generiert nur -14 bis +28, nicht älter (für history view)

**Falls nötig:** Appointments vor seed-dynamic.ts manuell in DB einfügen oder seed-dynamic.ts anpassen

### E2E tests fail mit "Data mismatch"

**Ursache:** seed-e2e-tests.ts data wurde verändert

**Lösung:**
```bash
npm run seed:e2e                # Re-run seed
npm run e2e-tests:ci            # Try tests again
```
